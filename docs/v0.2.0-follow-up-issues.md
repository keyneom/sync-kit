# Post-v0.2.0 sharing and concurrency follow-ups

These findings were verified against the Web TypeScript and Android Kotlin
implementations released as `0.2.0`. Keep behavior and tests in parity unless a
platform-specific limitation is explicitly documented and approved.

Release status (2026-07-16): the first six items are implemented with Web and
Android source/tests. The complete versioned `0.2.1` `npm run check` gate
passes, including Android, Java fixtures, JS/Kotlin parity, 157 TypeScript
tests, lint, typecheck, build, and packed-package verification. The remaining
items require an explicit API or policy decision and should not be folded into
an unrelated patch.

- [x] Required account-binding verifier guard
- [x] Fail-closed Drive write freshness
- [x] Reader-only ACL before link acceptance
- [x] Owner-only delete/trash authorization
- [x] Signed link file-manifest binding and exact validation
- [x] Cross-context protected-identity creation coordination
- [x] Full `npm run check`, including Android and parity fixtures

## Actionable fixes

### 1. Fail closed when required account binding has no verifier

**Problem:** The Drive-file acceptance path rejects this configuration, but
`acceptKeyResponseFromPayload` accepts a present binding without verifying it
when `requireAccountBinding` is true and no verifier is configured. This gives
consumers a false account-binding guarantee.

**Required fix:** On Web and Android, reject
`requireAccountBinding && verifyAccountBinding == null` before examining or
accepting the response. Apply the same invariant to every current and future
acceptance transport.

**Tests:** Cover missing verifier with both present and absent bindings for the
Drive-file and link-payload paths. No grant, ACL, registry head, or exchange
cleanup may change on failure.

### 2. Fail closed when Drive cannot prove write freshness

**Problem:** For a stored version that is not an HTTP ETag,
`assertWriteHeadFresh` only conflicts when the new metadata contains a
different `headRevisionId`. A missing `headRevisionId` is treated as fresh, and
the subsequent write uses the current HTTP ETag while publishing content based
on an older cryptographic revision. This can lose concurrent updates.

**Required fix:** When the stored version is not an HTTP ETag, require a present
and equal `headRevisionId`. Treat a missing token as a provider/state failure,
not as freshness. Preserve the existing conditional write and conflict-reread
behavior.

**Tests:** Add matching, mismatching, and missing-head cases on Web and Android.
The missing and mismatching cases must not issue a media write.

### 3. Do not grant Drive write access before cryptographic enrollment

**Problem:** `inviteParticipantForLink` immediately maps writer/admin grants to
a Drive writer permission. Before acceptance, that recipient is not in the
signed participant/grant set but can overwrite or delete the Drive object,
causing denial of service.

**Required fix:** At invitation time grant only Drive reader access, which is
sufficient for Picker discovery. Upgrade writer/admin recipients to Drive
writer only after the signed key response is verified and
`applyAcceptedGrants` has successfully published the enrolled envelope. Viewer
access remains reader. Do this on Web and Android.

**Tests:** Assert the invite-time and post-accept ACL sequence for viewer,
writer, and admin. Failed or unverified acceptance must never upgrade the ACL.

### 4. Authorize destructive dataset operations cryptographically

**Problem:** `deleteDataset` and `trashDataset` resolve a file ID from the
registry/listing and mutate Drive without reading the current envelope or
checking the caller's signed role. They rely entirely on provider ACLs.

**Required fix:** Read and verify the current owner-pinned head before either
operation, then require the current identity to be the cryptographic owner.
Keep deletion and trash behavior identical on Web and Android. Do not remove
the local registry record when verification or the provider mutation fails.

**Tests:** Owner succeeds; admin, writer, viewer, removed identities, invalid
heads, and owner-pin mismatches fail without provider mutation or registry
deletion.

### 5. Authenticate and strictly validate link dataset-file metadata

**Problem:** The invitation is signed, but the `sk-files` dataset/file/role map
is not. The recipient stores matching file IDs before proving the map is
complete, unique, or role-consistent. Envelope and owner-pin checks prevent
silent adoption of substituted content, but tampering can poison registry
routing and block onboarding.

**Implemented fix:** The canonical dataset/file/role manifest is SHA-256 hashed
into the invitation's existing signed `recipientDrivePermissionId` field using
a `link:` prefix. This avoids a new invitation schema while authenticating the
file IDs. Before any registry write, recipients require an exact one-to-one
match with the signed grants: no missing, duplicate, or unknown dataset IDs,
and identical roles. Legacy unbound `"link"` invitations are rejected by the
new recipient path and must be recreated.

**Tests:** Web/Kotlin fixtures and controller tests must reject missing, extra,
duplicate, role-mismatched, and file-ID-tampered manifests without changing the
registry. Preserve compatibility deliberately; do not change only one
platform's link wire format.

### 6. Make Web protected-identity creation conflict-safe

**Problem:** `getOrCreate()` performs a non-atomic load/create/IndexedDB `put`.
Two tabs can create different passkeys, overwrite the same stored record, and
leave one tab using an identity that is no longer persisted.

**Implemented fix:** Creation uses Web Locks across same-origin browser
contexts when available, and the IndexedDB store uses atomic
`saveIfAbsent`/`add` semantics as the final claim. Only the winner is cached and
returned. Custom stores can implement `saveIfAbsent`; older stores receive a
compatibility recheck but cannot promise atomic cross-device creation.
Document that an unused passkey credential may remain after a lost race.

**Tests:** Use two controllers and a barrier to force concurrent creation.
Exactly one persisted identity must win, and the loser must not cache or return
its discarded identity.

## Design and policy follow-ups

### Control-ledger poison recovery

Control payload merge is a synchronous event-ID union, while signature and
authority verification is asynchronous. A legitimate Drive writer can publish
an owner-signed envelope containing invalid control events and block later
control operations. Invalid events are not acted on, so this is an availability
and recovery problem rather than acceptance of forged control state.

Design a control-specific verified load/sync boundary that validates the
payload before updating a locally verified head or merging it into trusted
state. Define an owner recovery/repair procedure for replacing an invalid
remote payload while retaining conditional-write ancestry. Do not attempt to
call an asynchronous verifier from the existing synchronous codec merge.

### Manual Drive permissions during reconciliation

Permission reconciliation intentionally removes only permissions previously
tracked by sync-kit. Untracked manual shares therefore survive cryptographic
revocation and can continue downloading ciphertext, although rekeying prevents
decryption of new revisions.

Do not silently change the default to delete owner-managed Drive ACLs. If exact
ACL parity is required, add an explicit strict-reconciliation mode with audit
results and tests protecting owner/inherited permissions.

### Private snapshot cross-device concurrency

The private snapshot `CloudStore` contract has no revision token or
conditional-write operation, so simultaneous devices are last-write-wins even
though operations within one controller are serialized.

Document the current behavior. Any fix requires a versioned store contract and
Web/Android migration design; it is not a sharing-controller patch.

## Lower-priority cleanup

- Decide whether successfully accepted unbound exchange responses should be
  deleted; currently cleanup happens only after verified account binding.
- `CachingAuthorizationProvider.clear()` starts IndexedDB deletion without a
  completion signal. Add an asynchronous persistent-clear API if callers need
  a durable logout boundary.
- Keep provider response-body excerpts out of user-facing errors and logs, or
  expose them only through an explicitly diagnostic, redacted field.
- Constant-time comparison of public, one-time challenge strings is not a
  meaningful security priority.

## Do not implement these review suggestions as written

- Do not compare the recipient credential ID with the owner's protected
  identity. The owner does not possess the recipient's expected credential;
  durable continuity would require a separately pinned recipient enrollment.
- Do not add another JWKS cache or basic `iat`/`azp` checks. Both platforms
  already have bounded caches, unknown-`kid` refresh, and `iat`/`azp` checking.
  Optional future hardening is limited to policy such as `nbf` and an explicit
  maximum token age.
- Do not treat the fixed link permission marker as Drive provenance. Link
  responses are authenticated by their signatures and optional account
  binding; any invite-address matching rule is a separate consumer/account
  policy and should use a durable Google subject where available.
